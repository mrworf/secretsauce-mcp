# Milestone 06 Implementation Plan: Enrollment, Recovery, And Self-Service

## Scope review

Milestone 06 completes local human-credential ceremonies without adding the
Milestone 07 invitation/profile/status administration API. It may update the
host bootstrap command to issue the first temporary password, and it provides
guarded administrative reset domain operations for later routes, but it does not
publish an unguarded reset endpoint or implement service-assignment scope.

MCP OAuth/grants and durable gateway-reference ownership arrive in later
milestones. This milestone records one durable user-security invalidation event
inside every credential transaction, revokes current browser and restricted
sessions immediately, and defines a bounded post-commit invalidation sink.
Future grant/reference subsystems consume that contract rather than weakening
credential transactionality.

## Fixed configuration and secret contracts

The closed `identity` configuration gains `temporary_password_ttl`, defaulting
to 72 hours and bounded from one hour through seven days, plus
`restricted_session_ttl`, defaulting to 15 minutes and bounded from five through
30 minutes. Existing identity key files remain the only encryption/HMAC roots.

Temporary passwords are random base64url values of at least the current password
minimum and at least 24 characters. They are generated by the service, displayed
in exactly one non-cacheable response or terminal write, and stored only as the
existing bounded Argon2id encoding. They are never accepted from administrator
input, URLs, command arguments, logs, or audit fields.

Restricted session and CSRF values are independent random 32-byte base64url
values stored only as domain-separated HMAC-SHA-256 hashes. Their cookie is
`__Host-secretsauce_enrollment`, with `Secure`, `HttpOnly`, `SameSite=Strict`,
`Path=/`, and no `Domain`. The ordinary browser authenticator never accepts it.
Restricted routes revalidate purpose, expiry, user status/authenticator state,
user/global epochs, exact origin, and rotating CSRF on every use.

## Persistence and state-machine contract

Migration `0005` adds:

- one current temporary-password credential per user with hash, issue/expiry,
  purpose, version, and consumed/revoked timestamps;
- opaque restricted sessions with purpose
  (`initial_enrollment`, `password_change`, or `totp_enrollment`), hashes,
  epochs, expiry, and revocation;
- one pending TOTP ceremony per restricted session containing only an encrypted
  envelope plus creation/expiry state;
- durable user-security invalidation events containing user UUID, reason,
  event-time counts, creation time, and post-commit dispatch state.

No pending permanent password is stored. The begin-enrollment request accepts
and policy-validates the permanent password, creates/replaces the pending
encrypted TOTP ceremony, and returns that new seed and `otpauth` URI once.
Confirmation accepts the permanent password again, revalidates it against the
then-current policy, hashes it only for the final transaction, and commits it
with TOTP confirmation. This deliberately requires password-manager retention
through confirmation and avoids a durable unconfirmed password secret.

Confirmation uses an immediate transaction to recheck current policy version,
restricted-session state, temporary credential state when applicable, pending
envelope binding, and fresh TOTP step. It then installs the permanent password
and confirmed TOTP, consumes the accepted step, activates an invited or
enrollment-required identity, increments its security epoch, revokes all user
sessions/restricted sessions/temporary credentials, records invalidation, and
audits. Any failure leaves prior authenticators and status usable.

After administrative password reset, the existing TOTP is preserved. Temporary
login yields a `password_change` restricted session; completion requires the new
policy-compliant password plus the existing TOTP. After TOTP reset, permanent
password verification without TOTP may yield only a `totp_enrollment`
restricted session; it cannot create an ordinary session or authorize MCP.

## Restricted HTTP ceremonies

Public `POST /api/v2/auth/enrollment/login` accepts email and a temporary
password and returns only restricted-session metadata plus CSRF. Uniform
authentication failures cover nonexistent, expired, consumed, wrong, and
ineligible candidates with comparable Argon2 work.

Public `POST /api/v2/auth/totp-recovery/login` accepts email and the permanent
password only for an active local identity whose TOTP state is
`not_configured`; all other cases are uniform.

Restricted, same-origin, CSRF-protected routes are:

- `POST /api/v2/auth/enrollment/begin`: permanent password, returns one new
  base32 seed and `otpauth` URI;
- `POST /api/v2/auth/enrollment/confirm`: permanent password and new TOTP code,
  commits initial enrollment;
- `POST /api/v2/auth/password-change`: permanent password plus existing TOTP,
  commits a reset-required password change;
- `POST /api/v2/auth/totp-enrollment/begin` and `/confirm`: re-enroll after an
  administrative TOTP reset while preserving the permanent password.

Active ordinary browser sessions additionally expose:

- `POST /api/v2/auth/self/password`: current password, current TOTP, and new
  password;
- `POST /api/v2/auth/self/totp/begin`: current password and current TOTP,
  returns one new seed/URI and a restricted replacement ceremony;
- `POST /api/v2/auth/self/totp/confirm`: new TOTP confirmation, atomically
  replaces the authenticator and invalidates all state.

Every secret-entry route is `no-store`, declares secret fields, and uses closed
bounded schemas. QR rendering is client-side from the one-time `otpauth` URI;
the server stores or serves no QR image and provides no seed retrieval route.

## Administrative reset and invalidation contract

`LocalCredentialLifecycleService` exposes password-reset and TOTP-reset domain
methods. Each requires a non-empty bounded justification and an injected
authorization decision carrying the authenticated actor, target, capability,
and whether human step-up was satisfied. Browser admin/superadmin decisions
must carry step-up; permitted future API-key decisions are explicitly
non-human and never claim to satisfy it. Denied, missing, mismatched-target, or
missing-step-up decisions fail before mutation.

Password reset generates and returns one temporary password, preserves TOTP,
increments the user epoch, revokes all sessions/restricted sessions and prior
temporary credentials, records invalidation, and audits counts in one
transaction. TOTP reset returns no seed, deletes the authenticator, changes only
TOTP state, increments the epoch, performs the same invalidation, and audits.

The post-commit `IdentityInvalidationSink` receives only event ID, user UUID,
reason, and counts—never credential material. Dispatch failure cannot roll back
an already committed security mutation; the durable pending event remains
retryable and authorization paths continue to fail immediately through epochs.

## Break-glass contract

`npm run identity:break-glass` accepts no arguments and requires input and
output terminals. It prompts for a UUID or email and an exact confirmation,
generates rather than accepts a password, and prints the temporary value once
only after the atomic reset commits. The mutation preserves UUID and role,
places the existing identity into `enrollment_required`, deletes TOTP, increments
the epoch, revokes all user state, and records actor type `local_cli`, stable
host-terminal authentication, bounded operating-system actor metadata when
available, justification, outcome, and invalidation counts.

The command has one stable target-not-found/reset failure and never prints the
submitted identifier, configuration/database/key paths, or internal errors.
Because no role is removed and recovery restores enrollment access, the
last-active-superadmin invariant is not weakened.

The existing bootstrap CLI atomically creates the initial pending superadmin and
its temporary credential, then displays that generated value once. It still
accepts no password/TOTP argument or prompt.

## Slice 1: configuration, migration, temporary credentials, and invalidation

Outcome: bounded configuration; migration `0005`; random temporary-password
generation; restricted-session hashing; transactional invalidation records; and
guarded administrative password/TOTP reset primitives.

Positive tests cover exact TTL bounds, generated length at minimum 8/128, hash
only persistence, TOTP-preserving password reset, seed-erasing TOTP reset,
epoch/session revocation, justification audit, and invalidation sink delivery.
Negative tests cover unknown config fields/out-of-range TTLs, caller-supplied
password attempts, missing/oversized justification, denied/missing step-up,
unknown target parity, rollback on audit/database failure, and absence of every
secret from records/errors/audit.

Commit: `Add guarded local credential resets`.

## Slice 2: restricted authentication and initial enrollment

Outcome: uniform temporary/recovery login, isolated restricted cookies and
CSRF, one-time TOTP begin response, atomic initial enrollment/password-change/
TOTP-enrollment completion, and bootstrap temporary-password issuance.

Positive tests cover first bootstrap enrollment, reset-required password change,
TOTP-reset re-enrollment, expiry edge, policy-version recheck, restart, and one
winning concurrent confirmation. Negative tests cover ordinary login/MCP use,
expired/reused temporary passwords, wrong credentials/codes, missing
confirmation, seed retrieval, altered cookie/CSRF/origin, ceremony mismatch,
and transaction rollback retaining prior state.

Commit: `Complete restricted local enrollment`.

## Slice 3: authenticated self-service replacement

Outcome: browser-session password change and two-step TOTP replacement routes
using current password plus fresh current TOTP, with response-complete logout
and durable invalidation.

Positive tests cover all roles, password replacement, new-seed confirmation,
old/new authenticator behavior, and current/all-session revocation. Negative
tests cover API keys, wrong/replayed authenticators, changed session/user/epoch,
weak/context/compromised new passwords, unconfirmed seed, expired ceremony,
concurrent replacement, audit failure, and proof/secret absence.

Commit: `Add local security self-service`.

## Slice 4: break glass, lifecycle, documentation, and acceptance

Outcome: terminal-only break-glass CLI, bootstrap handoff updates, sanitized
readiness/shutdown checks, operator documentation, and the M06 acceptance audit.

Positive tests cover UUID/email target selection, all account states/roles, OS
actor metadata, one-time output, restart, and idempotent close. Negative tests
cover arguments/non-TTY/cancel/not-found, user-selected-password attempts,
output failure, audit failure, unsafe config/key state, and output/log/audit
secret inspection.

Acceptance runs focused migration/config/credential/restricted-session/control/
CLI tests, production build, generated OpenAPI consistency, `git diff --check`,
and the unchanged full suite.

Commit: `Complete local credential lifecycle`.

## Later-milestone handoff

Milestone 07 adds invitation and reset routes around the guarded domain methods,
including related-service scope and role matrices. Milestones 13–15 attach
runtime-reference and OAuth grant consumers to durable invalidation events.
Milestone 18 adds settings UX and global password/TOTP events without changing
the per-user transaction contract.
